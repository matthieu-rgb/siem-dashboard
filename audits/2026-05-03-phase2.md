## Findings by Severity

### CRITICAL

---

#### PHASE2-AUTH-001

- **Severity:** Critical
- **Component:** Backend - `app/core/config.py`
- **Title:** Hardcoded weak default secret key with no enforcement

**Description:**

`settings.secret_key` defaults to the literal string `"change-me-in-production"`. There is no runtime guard that prevents the application from starting with this value. Any deployment where `.env` is missing or incomplete will silently use this key.

```python
secret_key: str = "change-me-in-production"  # noqa: S105
```

The `# noqa: S105` suppression actively hides the Bandit/ruff-S warning, meaning CI/CD pipelines with static analysis will not catch it.

**Impact:**

An attacker who knows the default key (it is visible in the public source) can forge arbitrary JWTs for any `sub` value, effectively gaining access as any user including the admin. This is a full authentication bypass.

**Remediation:**

1. Remove the default value entirely and make the field required, or validate at startup:

```python
from pydantic import field_validator

@field_validator("secret_key")
@classmethod
def secret_key_must_not_be_default(cls, v: str) -> str:
    if v in ("change-me-in-production", "", "secret"):
        raise ValueError("SECRET_KEY must be set to a strong random value in production")
    if len(v) < 32:
        raise ValueError("SECRET_KEY must be at least 32 characters")
    return v
```

2. Generate a proper key: `python -c "import secrets; print(secrets.token_hex(32))"`.
3. Remove the `# noqa` suppression so static analysis catches any future regression.

#### PHASE2-AUTH-002

- **Severity:** Critical
- **Component:** Backend - `app/api/auth.py`
- **Title:** Refresh token rotation without revocation - stolen tokens remain valid for 7 days

**Description:**

The `/refresh` endpoint issues a new refresh token on every call but never invalidates the old one. There is no server-side token store, no jti (JWT ID) claim, and no revocation list. The refresh token has a 7-day lifetime (`refresh_token_expire_days: int = 7`).

```python
# auth.py - /refresh endpoint
_set_refresh_cookie(response, create_refresh_token(sub))   # new token issued
# old token: still valid until its exp claim, no invalidation
```

Logout only deletes the cookie client-side:

```python
async def logout(response: Response) -> dict[str, bool]:
    response.delete_cookie(key=_REFRESH_COOKIE, path="/api/auth", samesite="lax")
    return {"ok": True}
```

**Impact:**

- If a refresh token is exfiltrated (XSS, network interception, compromised device), the attacker retains a valid 7-day session even after the legitimate user has logged out or rotated.
- A logout does not invalidate the session server-side. An attacker with the old cookie can immediately re-authenticate.

**Remediation:**

1. Introduce a `refresh_tokens` table with columns `(jti UUID PK, user_id FK, expires_at, revoked_at)`.
2. Add a `jti` claim to every refresh token at issuance.
3. On `/refresh`: look up the jti, verify it is not revoked, then revoke it before issuing the new one (atomic swap).
4. On `/logout`: mark the current jti as revoked.
5. Optionally implement a family-based revocation: if a jti is reused (replay), revoke all tokens for that user.

#### PHASE2-AUTH-003

- **Severity:** High
- **Component:** Backend - `app/api/auth.py`
- **Title:** Refresh token cookie missing `Secure` flag

**Description:**

The `_set_refresh_cookie` helper explicitly sets `secure=False`:

```python
response.set_cookie(
    key=_REFRESH_COOKIE,
    value=token,
    httponly=True,
    samesite="lax",
    secure=False,          # <-- plaintext HTTP allowed
    max_age=_COOKIE_MAX_AGE,
    path="/api/auth",
)
```

**Impact:**

The refresh token will be transmitted over unencrypted HTTP connections. Any network observer (coffee shop MITM, corporate proxy, Suricata on the path) can capture it and reuse it within the 7-day window. This is particularly relevant given the HomeLab context where the dashboard may be accessed over LAN without TLS.

**Remediation:**

Set `secure=True`. Use an environment-aware flag if you need local HTTP dev access:

```python
secure=settings.cookie_secure   # default True, set False only via .env in dev
```

The `SameSite="lax"` value is appropriate but should be tightened to `"strict"` if the refresh endpoint is never used via cross-site top-level navigations (it is not - it is a POST-only API route).

#### PHASE2-AUTH-004

- **Severity:** High
- **Component:** Backend - `app/core/rate_limit.py`
- **Title:** Rate limiter keyed on `request.client.host` - trivially bypassed behind a reverse proxy

**Description:**

The login rate limiter keys on `request.client.host`:

```python
ip = request.client.host if request.client else "unknown"
if not login_limiter.is_allowed(ip):
```

When the application runs behind Nginx, Caddy, or any reverse proxy (which is the case here - Nginx on the Raspberry Pi), `request.client.host` is the proxy's IP address, not the client's. All login attempts from every real user share one bucket, making the limiter both:

- **Ineffective**: an attacker probing from real IPs does not get rate-limited.
- **DoS-capable**: 5 login attempts from any legitimate user exhaust the shared bucket and lock out every other user for 15 minutes.

Additionally, the limiter is in-memory and per-process. Under multiple workers (`uvicorn --workers N`) each worker has its own independent counter, reducing the effective limit to 5 * N per IP.

**Impact:**

Credential stuffing / brute force attacks are not blocked. An attacker with a botnet of N IPs can attempt 5*N passwords before any IP is limited. A single user (or attacker) can deny login to all users for 15 minutes with 5 requests through the proxy.

**Remediation:**

1. Use `X-Forwarded-For` or `X-Real-IP` headers with a trusted proxy allowlist. FastAPI/Starlette supports this via `ProxyHeadersMiddleware` with a `trusted_hosts` list:

```python
from uvicorn.middleware.proxy_headers import ProxyHeadersMiddleware
app.add_middleware(ProxyHeadersMiddleware, trusted_hosts=["127.0.0.1", "192.168.100.248"])
```

2. Replace the in-memory limiter with a Redis-backed counter (e.g. `slowapi` with Redis backend) to handle multi-worker deployments.
3. Consider keying on `(ip, username)` to avoid cross-user DoS while still limiting per-account brute force.

#### PHASE2-AUTH-005

- **Severity:** High
- **Component:** Backend - `app/core/config.py` / `app/main.py`
- **Title:** OpenAPI documentation endpoints exposed with no authentication

**Description:**

The FastAPI application exposes Swagger UI and ReDoc at publicly accessible paths:

```python
app = FastAPI(
    docs_url="/api/docs",
    redoc_url="/api/redoc",
    openapi_url="/api/openapi.json",
    ...
)
```

There is no middleware, dependency, or IP restriction protecting these endpoints.

**Impact:**

For a SIEM dashboard (security-sensitive tool), the API schema exposes all endpoint signatures, request/response models, and authentication mechanisms to unauthenticated users. This assists attackers in mapping the attack surface without any prior knowledge. In a production deployment this is an information disclosure issue.

**Remediation:**

Disable in production or restrict with a dependency:

```python
docs_url="/api/docs" if settings.debug else None,
redoc_url="/api/redoc" if settings.debug else None,
openapi_url="/api/openapi.json" if settings.debug else None,
```

#### PHASE2-AUTH-006

- **Severity:** High
- **Component:** Backend - `app/api/auth.py`
- **Title:** Rate limiting applied after DB query on login - timing oracle on email enumeration

**Description:**

The login endpoint executes the database query before checking whether the result would be a valid user, and returns distinct HTTP status codes for rate limit (429) vs invalid credentials (401). However, the more subtle issue is that the rate limiter calls `is_allowed()` and records a hit regardless of whether the credentials were valid. The actual enumeration vector is that the rate limiter check happens before the DB query:

```python
if not login_limiter.is_allowed(ip):        # 429 if over limit
    raise HTTPException(status_code=429, ...)

result = await db.execute(select(User).where(User.email == body.email))
user = result.scalar_one_or_none()

if user is None or not verify_password(body.password, user.password_hash):
    raise HTTPException(status_code=401, detail="Invalid credentials")  # same for both

if not user.is_active:
    raise HTTPException(status_code=403, detail="Account disabled")  # different code!
```

The `403 "Account disabled"` branch is reachable only when the email exists AND the password is correct AND `is_active=False`. This leaks the existence of a valid account with a known password. Separately, `argon2id` verification has a measurable time cost (~50-100ms) - when `user is None` this cost is skipped entirely, creating a timing side-channel to enumerate valid email addresses.

**Impact:**

An attacker can enumerate valid email addresses via response timing. An attacker with valid credentials for a disabled account gets a distinct response, confirming account existence and password correctness.

**Remediation:**

1. Apply a constant-time dummy hash verification when the user is not found:

```python
if user is None:
    _pwd_context.dummy_verify()  # passlib supports this
    raise HTTPException(status_code=401, detail="Invalid credentials")
```

2. Return 401 for disabled accounts instead of 403 (or always return the same 401 message for all auth failures).

#### PHASE2-AUTH-007

- **Severity:** Medium
- **Component:** Backend - `app/core/security.py`
- **Title:** Algorithm confusion risk - `algorithms` parameter accepts only HS256 but no explicit rejection of asymmetric algorithms at decode

**Description:**

`decode_token` passes `algorithms=[ALGORITHM]` to `jwt.decode()` which is correct. However, the `ALGORITHM = "HS256"` constant is a module-level string with no validation. If a developer changes it to `"RS256"` or `"none"` without corresponding key material changes, tokens will fail silently at verification or be accepted unsigned. The `python-jose` library has a history of `alg: "none"` vulnerabilities in older versions.

**Impact:**

Low exploitability in isolation, but contributes to future vulnerability if the algorithm is ever changed carelessly. The `alg: "none"` attack is not exploitable here because `algorithms=[ALGORITHM]` is passed to decode, but only if the library version is not vulnerable to header injection bypasses.

**Remediation:**

1. Pin the algorithm to an explicit allowlist with an assertion:

```python
_ALLOWED_ALGORITHMS = frozenset({"HS256"})
assert ALGORITHM in _ALLOWED_ALGORITHMS
```

2. Verify that `python-jose` is up to date - versions before 3.3.0 had `alg: "none"` issues.
3. Consider migrating to `PyJWT` which has a cleaner security track record for this library surface.

#### PHASE2-AUTH-008

- **Severity:** Medium
- **Component:** Backend - `app/core/dependencies.py`
- **Title:** `int()` conversion of JWT `sub` claim without bounds checking

**Description:**

```python
result = await db.execute(select(User).where(User.id == int(user_id_raw)))
```

`int(user_id_raw)` raises `ValueError` if the string is not a valid integer, which is not caught and will produce an unhandled 500 error rather than a 401. More importantly, there is no upper-bound check, so a crafted token (if the secret is compromised) with `sub: "99999999999999999"` will issue a valid DB query that returns no user, which is handled correctly (returns 401), but the unguarded `int()` conversion is not.

**Impact:**

With a valid secret, an attacker cannot exploit this further than normal token abuse (already covered by PHASE2-AUTH-001). Without a compromised secret, the `ValueError` path would produce a 500 and leak a stack trace if FastAPI's debug mode is enabled.

**Remediation:**

```python
try:
    user_id = int(user_id_raw)
except ValueError:
    raise HTTPException(status_code=401, detail="Malformed token")
result = await db.execute(select(User).where(User.id == user_id))
```

#### PHASE2-AUTH-009

- **Severity:** Medium
- **Component:** Backend - `app/main.py`
- **Title:** CORS `allow_headers=["*"]` with credentials enabled

**Description:**

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.cors_origins,
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

`allow_credentials=True` combined with `allow_headers=["*"]` is technically invalid per the Fetch specification (browsers reject `Access-Control-Allow-Headers: *` with credentials), but FastAPI/Starlette silently accepts this configuration. The practical result is that preflighted requests with custom headers may behave unexpectedly across browsers.

More importantly, `cors_origins` is a configurable list with no runtime validation. If an operator mistakenly sets `cors_origins=["*"]`, the combination with `allow_credentials=True` would be a critical CORS misconfiguration. Starlette does warn about this, but silently in some versions.

**Impact:**

Current default (`localhost:8080`, `localhost:5173`) is safe. The risk is configuration drift - no guard prevents a wildcard origin from being added.

**Remediation:**

1. Enumerate allowed headers explicitly instead of using `*`: `allow_headers=["Authorization", "Content-Type"]`.
2. Add a validator in `config.py`:

```python
@field_validator("cors_origins")
@classmethod
def no_wildcard_cors(cls, v: list[str]) -> list[str]:
    if "*" in v:
        raise ValueError("Wildcard CORS origin is not permitted with credentials")
    return v
```

#### PHASE2-AUTH-010

- **Severity:** Medium
- **Component:** Frontend - `src/api/client.ts`
- **Title:** `api.auth.refresh()` in client.ts does not send `credentials: 'include'`

**Description:**

The `api.auth.refresh()` export uses `apiFetch` (not `authFetch`):

```typescript
refresh: () =>
  apiFetch<TokenResponse>('/auth/refresh', {
    method: 'POST',
    credentials: 'include',   // <-- this is inside init.headers? No - it is RequestInit
  }),
```

Wait - this is passed as part of `init` to `apiFetch`, which spreads `...init` into the `fetch` call. So `credentials: 'include'` is correctly forwarded. However, `apiFetch` also explicitly spreads `headers` before `...init`, meaning the `headers` inside `init` will override `Content-Type`. The actual risk here is in `AuthContext.tsx`:

```typescript
const { access_token } = await api.auth.refresh()
```

This call at page load goes through `apiFetch` which does not include the access token. If the `/refresh` endpoint ever requires authentication (e.g., after adding an auth requirement), this silently fails. The intent (unauthenticated refresh via cookie) is correct, but there is no test coverage ensuring the cookie is sent, and the `credentials: 'include'` flag in `apiFetch` is controlled entirely by the caller passing it in `init` - the base function does not force it.

**Impact:**

Current code works correctly. The risk is that future changes to `apiFetch` (e.g., removing `...init` spreading) could silently drop the credentials flag and break token refresh without an obvious failure mode.

**Remediation:**

Create a dedicated `apiFetchWithCredentials` helper or enforce `credentials: 'include'` in `apiFetch` globally, since this API always runs same-origin via the Vite proxy.

#### PHASE2-AUTH-011

- **Severity:** Low
- **Component:** Backend - `app/services/bootstrap.py`
- **Title:** Bootstrap password printed to stdout in plaintext

**Description:**

```python
print(f"  Initial pwd   : {password}   (change immediately)")
```

The bootstrap password is printed to stdout. In a containerized environment this ends up in Docker logs, which may be collected by a log aggregator, stored in plaintext, and accessible to anyone with log read access.

**Impact:**

If Docker logs are shipped to a SIEM (including Wazuh), the initial admin password appears in indexed log data. Low severity because the password is one-time and a prompt to change it is shown, but log retention means it may persist.

**Remediation:**

Use `structlog` (already imported) with a masked field, or write to stderr and immediately clear the variable after printing. Consider writing it to a file with `chmod 600` instead of stdout.

#### PHASE2-AUTH-013

- **Severity:** Info
- **Component:** Backend - `app/core/security.py`
- **Title:** argon2id configuration relies on passlib defaults

**Description:**

`CryptContext(schemes=["argon2"], deprecated="auto")` uses passlib's defaults for argon2id parameters. As of passlib 1.7.x, the defaults are: time_cost=2, memory_cost=102400 (100 MB), parallelism=8, hash_len=16. These are reasonable but not aligned with the OWASP 2024 recommendation (time_cost=3, memory_cost=65536 minimum, or time_cost=1, memory_cost=2097152 for high-security contexts).

**Impact:**

Defaults are acceptable for most deployments. No direct vulnerability.

**Remediation:**

Explicitly configure the parameters to document the security decision and prevent silent changes if passlib changes its defaults:

```python
_pwd_context = CryptContext(
    schemes=["argon2"],
    deprecated="auto",
    argon2__time_cost=3,
    argon2__memory_cost=65536,
    argon2__parallelism=1,
    argon2__hash_len=32,
)
```

#### PHASE2-AUTH-015

- **Severity:** Info
- **Component:** Backend - `app/api/auth.py`
- **Title:** Audit log records IP but not user-agent or failed login attempts

**Description:**

`AuditLog` is created on successful login with `target=ip`. Failed login attempts are not logged at all.

**Impact:**

Reduces forensic value of the audit log. A brute force attack that eventually succeeds logs only the success. The SIEM dashboard itself has blind spots in its own audit trail.

Log failed authentication attempts with `action="auth.login_failed"` and include the attempted email (hashed or masked) and user-agent in the `extra` JSON field.

## Positive Findings

The following were explicitly checked and found to be correctly implemented:

- **Token type enforcement**: Both `/refresh` and `get_current_user` verify the `type` claim against the expected constant. An access token cannot be used as a refresh token and vice versa.
- **httpOnly on refresh cookie**: The refresh token is never accessible to JavaScript.
- **Cookie path scoping**: `path="/api/auth"` limits the cookie to auth endpoints, reducing the attack surface.
- **Password hashing**: argon2id is used correctly. No MD5, SHA-1, or bcrypt with low cost.
- **RBAC structure**: `get_current_admin` and `get_current_analyst_or_above` are clean dependency-injection guards. No role check relies on client-supplied data.
- **ORM usage**: All DB queries use SQLAlchemy ORM/select constructs with bound parameters. No raw SQL string interpolation. SQL injection surface is minimal.
- **Access token in memory**: The access token is stored in a module-level variable (`_accessToken`), not in `localStorage` or `sessionStorage`. It does not survive page reload (by design, relying on cookie-based refresh).
- **Bootstrap entropy**: `secrets.choice()` over 66-character alphabet for 24 characters gives ~143 bits of entropy. Adequate.
- **`is_active` check**: Both login and `get_current_user` verify the `is_active` flag, preventing soft-deleted users from authenticating.
